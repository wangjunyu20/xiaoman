## 错误处理原则

### 发现错误时
1. **立即停止** — 不要继续生成更多错误内容
2. **定位问题** — 找出根本原因（流程/数据/代码）
3. **修复并验证** — 修复后必须重新检验
4. **记录复盘** — 写入 memory/ 防止再犯

### 用户反馈问题时
1. **承认问题** — 不找借口，不推诿
2. **立即检查** — 验证问题是否存在
3. **快速修复** — 优先解决问题而非解释
4. **预防再犯** — 更新流程防止重复发生

## 报告生成铁律（不可违背）

### 1. 部署流程铁律
- **必须一键部署** — 使用 `deploy_full.sh`，禁止分步手动执行
- **必须通过检验** — `verify_reports.py` 失败 = 禁止推送
- **必须双格式输出** — 网页版 + PDF版（带目录）
- **必须验证链接** — 推送后点击至少3个详情页验证可访问
- **禁止推送空文件** — 详情页<500字符视为空，禁止推送

### 2. 行业管理铁律
- **禁止重复定义** — 同一产业只能有一个行业（如"具身智能/人形机器人"）
- **新增需审核** — 变更行业清单必须检查是否重复
- **定期梳理** — 每季度审查行业清单，合并重复项

### 3. 数据质量铁律
- **禁止编造数据** — 所有数据必须真实可查
- **充分查询** — 使用所有可用工具获取数据
- **多源验证** — 关键数据必须交叉验证
- **标注来源** — 每条数据必须标明来源和时间

### 4. 模板规范铁律
- **严格按模板填充** — 不得遗漏任何栏目
- **首页+详情页+PDF** — 完整输出，缺一不可
- **PDF带目录** — 必须包含首页+所有行业详情页
- **跳转链接** — 首页必须能跳转到详情页
- **移动端适配** — 所有页面必须支持手机浏览

### 5. 持续完善原则
- **只能增量，不能减量** — 每次更新必须比上次更丰富
- **只能前进，不能回退** — 不断完善，绝不缩水
- **严格按SKILL.md执行** — 模板要求的栏目必须全部填充

## 需求管理铁律（新增 2026-03-11）

### 任务执行流程
**当用户提出需求或修改意见时，必须执行以下流程，不得擅自停止：**

```
1. 立即记录需求
   → 创建/更新 memory/requirement_<任务名>.md
   → 记录原始需求、用户反馈、修正意见

2. 确认理解正确
   → 向用户复述需求，确认理解无误
   → 明确完成标准

3. 持续迭代实现
   → 每次修改后主动汇报进展
   → 邀请用户验证
   → 记录用户反馈

4. 直到用户确认
   → 需求完成 ≠ 我觉得可以了
   → 需求完成 = 用户明确说"可以了"/"达到要求了"
   → 未完成的需求不得标记为完成

5. 归档复盘
   → 完成后更新需求文档，标记完成
   → 记录经验教训
```

### 禁止行为
- ❌ 自己觉得可以了就停止
- ❌ 不记录用户反馈
- ❌ 不确认完成标准就动手
- ❌ 回避用户的纠正意见

### 强制检查清单
每次用户反馈后，必须检查：
- [ ] 需求文档是否更新？
- [ ] 用户反馈是否记录？
- [ ] 当前进展是否汇报？
- [ ] 用户是否确认完成？
